Refactor -- Treatment

发表于 2019-08-25 更新于 2020-06-07

重新组织函数

  1. Extract Method — Inline Method
  2. Inline Temp (内联临时变量)
  3. Replace Temp with Query (以查询取代临时变量)
  4. Introduce Explaining Variable (引入解释性变量)
  5. Split Temporary Variable (分解临时变量)
  6. Remove Assignments to Parameters (移除对参数的赋值)
    • 即对参数的修改是在副本发生的, 不影响原数据
  7. Replace Method with Method Object (以函数对象取代函数)
    • strategy pattern
  8. Substitute Algorithm (替换算法)(优化逻辑)

在对象之间搬移特性

  1. Move Method
  2. Move Field
  3. Extract Class — Inline Class
  4. Hide Delegate (隐藏委托关系) — Remove Middle Man (移除中间人)
  5. Introduce Foreign Method (引入外加函数)
    • 缩短或隐藏参数列表的一种手段
    • 通过将 this 等作为参数传入
  6. Introduce Local Extension (引入本地扩展)
    • 子类化或包装类

重新组织数据

  1. Self Encapsulate Field (自封装字段)
    • getter / setter
  2. Replace Data Value with Object (以对象取代数据值)
  3. Change Value to Reference (将值对象改为引用对象) — Change Reference to Value (将引用对象改为值对象)
    • 工厂模式
  4. Change Unidirectional Association to Bidirectional(将单向关联改为双向关联) — Change Unidirectional Association to Bidirectional(将双向关联改为单向关联)
  5. Replace Magic Number with Symbolic Constant (以字面常量取代魔法数)
  6. Encapsulate Collection (封装集合)
  7. Replace Record with Data Class (以数据类取代记录)

Refactor -- Code Smell

发表于 2019-08-18 更新于 2019-08-24 分类于 Refactor

Code Smell in Method

  1. Long Method
  2. Duplicated Code
  3. Long Parameter List
  4. Switch Statements
  5. Feature Envy (依恋情节)
    • 函数对某个类的兴趣高过对自己所处类的兴趣. 通常是兴趣点是数据
  6. Middle Man (中间人)
  7. Comments (过多的注释)

Code Smell in Class

  1. Large Class
  2. Temporary Field (令人迷惑的暂时字段)
    • 类中的某些字段只有在特殊时候才会有用, 满足某些特殊算法
  3. Divergent Change (发散式修改)
    • 类经常因为不同的原因在不同的方向上发生变化.
  4. Shotgun Change (霰弹式修改)
    • 每遇到某种变化, 你都必须在许多不同的类内部做出许多小修改.
  5. Lazy Class (冗赘类)
    • 价值不大的类?

Code Smell between Classes

  1. Data Clumps (数据泥团)
    • 两个类中的相同的字段, 许多函数签名中相同的参数. 这些总是绑在一起出现的数据应该拥有属于它们自己的对象
    • 删除众多数据中的一项后, 其他数据有没有因此而失去意义
  2. Inappropriate intimacy
    • 两个类过于关注彼此之间的 private 成分

Code Smell about Extending

  1. Parallel Inheritance Hierachies (平行继承体系)
    • 每当你为某个类增加一个子类, 必须也为另一个类增加一个子类
  2. Refused bequest (被拒绝的遗赠)
    • 子类只需要父类中某些字段或某些方法.
    • 可通过在子类和父类之间再添加一个超类来解决

Others

  1. Primitive Obsession (基本类型偏执)
  2. Speculative Generality (夸夸其谈未来性)
  3. Alternative classes with different interfaces (异曲同工类)
  4. Incomplete library class (不完美的类库)

Concurrent

发表于 2019-08-12 更新于 2019-08-18

volatile

在 JVM 中, 如果A线程与B线程修改同一个类变量(class filed), 这一个修改的操作, 在A线程与B线程运行期间是互相之间是不可见的. 这是因为, 线程在执行的时候会从主内存获取最新的变量值, 然后之后在最后执行完毕的时候, 才会将工作内存中的内存刷新回去主内存. 这种特性我们称之为线程之间的 不可见性.

但是在 JAVA 中, 你可以使用 volatile 来强制线程每一次对类变量的修改, 都会立刻将该值刷新回去主内存.

volatile 虽然消除了线程之间的 不可见性, 但是它并不能保证线程是线程安全的.

CAS(Compare and Set)

CAS, 可以看作是乐观锁的一种实现. 它分为三个操作

  1. 首先获取 old value(当前值)
  2. 接着对 old value 进行自增获得 expect(期望值)
  3. 最后使用 old value 与当前内存中 new value(最新值) 进行比较
    1. 如果结果相等, 说明没有线程操作过该值, 则将 expect 对内存中的设置
    2. 如果不等, 则从步骤一重新开始

在 JAVA 中对应的就是 Atomic. 它是通过 JNI 调用CPU指令来执行的 CAS. 但是它有一个问题, 就是如果有大量的线程同时并发的修改该值, 那么就会有大量的线程不断的循环步骤123. 所以在 JAVA 中通过使用 分段CAS 以及 自动分段迁移 对其进行了优化, 对应的类是 LongAdder

其内部是这样的 内部会搞一个Cell数组, 每个数组是一个数值分段. 这时, 线程会分别去对不同Cell内部的value值进行CAS操作, 如果你想获取最后的值, 那么它就会将不同的Cell累加返回给你.

AQS(AbstraQueueSynchronize)

ReentrantLock 内部就是使用 AQS 来进行实现的. AQS 中拥有两个变量, 一个是 state 保存当前加锁数量和状态, 一个是 this 保存当前锁持有者.如果一个线程要获取当前的锁

  1. 那么它需要判断 state 是否为0,
  2. 若为0, 则说明还没有人获得该锁
  3. 接着使用 CAS 对 state 加一, 这就获取锁成功了.
  4. 释放锁的时候, 对应的就是自减

而重入锁的时候就是, 多出一步判断 this 对象是不是自己. 若是, 则使用 CAS 进行自增.

而公平的可重入锁就是, 在其它线程获取锁失败的时候, 会进入 AQS 中的一个等待队列. 而每一次释放锁之后的线程唤醒, 就会对等待队列中的第一个进行唤醒.

something about ReadWriteLock

读写锁适用场景为读多写少的场景.

Microservice

发表于 2019-08-11 更新于 2019-08-18

Service discovery

在 microservice 中 service 的注册与发现是非常重要的一个环节. 同一种类的 service 对应的是部署在不同机器上的 application, 通过向 service discovery center(服务发现中心) 进行注册, 被其他 service 获知.

例如 customer service 有3个 application 在不同的机器上. customer service 在 service discovery center 使用 customer 这个 serviceID 进行注册之后, 这个 customer service 对应的3个 application 的IP和端口就可以通过 serviceID (即 customer )暴露在这个集群中.

这里比较巧妙的地方在于使用 serviceID 屏蔽了复杂且会不断变动的 ${hostname:port} 列表. 同时充当了一个 Health Indicator(健康检测) 的作用(长时间未发送 hearthbeat 的 application 将会从注册列表中移除).

Circuit Breaker

circuit breaker(断路器) 在 microservice 中起到保护整个系统的可靠性的作用. 试想这样一个场景, 如果你的服务A挂了, 然后服务B依赖于服务A, 这个时候你的服务B也会因为服务A挂而挂.

而 circuit breaker 提供了一种名为 isolation(隔离) 的机制, 它使用多个线程池保证只有与调用服务A相关的线程池会阻塞.

然后提供了一种名为 fuse(熔断) 的机制, 使得对已经挂掉的服务不会再去执行超时重试的操作, 而是直接返回.

然后 Service downgrade(服务降级) 的机制, 来对本次失败操作进行补偿, 例如保存当前失败操作的信息等.

Client Side Load Balancer

在 microservice 中 service 之间都是通过向 service discovery center 主动拉取 service 对应的 IpAddres List 来进行访问的. 我个人理解是出于对 service discovery center 请求压力的缓解, 所以采用了 client side load balancer(客户端负载均衡) 的策略

service discovery center 每天在 microservice 中承受的压力是巨大的. 每个 application 每隔30s就会发送自己的 heartbeat(心跳) 到 service discovery center.

client 只需要简单地从 service discovery center 拉取 service 对应的 IpAddres List, 然后使用最简单 Round Robin(轮询) 策略即可.

API gateway

可以简单理解为就是 nginx, 提供 Reverse proxy(反向代理) 的功能. 同时你还可以在这里通过 Filter(拦截器) 去做一些其他的操作. 而为什么使用 Netflix 中的 Zuul 而不使用 nginx 作为 API gateway. 我个人认为是因为 Zuul 能很好的利用 Netflix 体系中的其他组件对 API gateway 进行开发.

作为一个统一的系统入口, 它可以做到 Authorization(授权), Limiting(限流), log(日志) 等操作.

Config Center

如果你的一个 service 对应有十几个甚至几十个 application, 那么你想修改这个 service 所对应的 application setting file 的时候, 这将会一件非常耗时的操作.

所以在 microservice 中引入了 config center(配置中心的概念). 通过它你可以将某一个 service 对应的 config file 移动至一个统一的位置进行管理, 这个地方可以是某一个 Git仓库.

而且通过和 endpoints 结合还可以做到 hot update, 不过还需要一个 Bus 去对全局的 application 发送拉取最新配置的事件.

Message Queue

Message Queue(消息队列) 早在传统的 SOA 项目中就已经存在. Message Queue 在系统可以充当服务调用的媒介, 但同时它最重要的两个作用是 柔性事务 和 流量削峰.

Message Queue 是 Event Drive 模型的基础, 也就是说只有 A service 产生了一份数据, B service 才会被驱动执行的相应的逻辑. 这里与 RPC 最主要的区别在于, 如果 A service 在运行过程中需要调用 B service, 那么 Event Drive 就没办法满足这种需求.

RPC 适合在 XA(二阶段提交) 的场景中使用.
Event Drive 适合在追求 最总一致性 的场景中使用.

Kubernetes

发表于 2019-08-10 更新于 2019-08-18

Pod

kubernets 的最小部署单位. 抽象级别的操作系统. 定义了一组互相依赖的容器, 它们之间可以通过 localhost 或者是本地文件的方式进行数据交互.

这个抽象的出现是为了解决一组强依赖的部署对象. 例如, 日志应用, 普通应用之间就属于一种强依赖, 一般我们希望将这两个应用部署在同一台物理机器上. 而 Pod 的引入就是为了解决这个问题.

Service

访问集群中的 Pod 的一层网络代理, 具有负载均衡的功能.

Pod 在集群中的 IP 随着 Pod 的更新等操作, 不会一直绑定一个值. 这个时候就需要对外提供一层不变的 IP 供其它的 Pod 或者用户访问对应的 Pod. 可以暂时简单理解为对 Pod 做了一层反向代理.

Volumes

与 Docker 中的 Volumes 类似. 只是在 Kubernetes 中 Volumes 是与 Pod 绑定的. 且支持的类型更加丰富

Master & Node

  • Master 指的是 kubernetes 中的调度节点.
  • Node 指的是 kubernetes 中的物理节点.

Type of Vlomes

configMap

类似于配置中心. 通过文件将保存在 kubernetes 中的 key-value 注入 container 中.

secret

私有安全信息. 将保存在 kebernetes 中对应的安全信息通过文件挂载的方式注入 container. (注意, 保存在 kubernetes 中的 secre 信息必须是经过加密的.)

others

  • minikube, kubeadm 不可用于生产环境中.

Problems

  1. 在集成了 Spring Security 的 Spring Boot 项目进行 Controller 的测试

这样做可以避开 Spring Security 专注于测试 Controller 的 Mapping. 又或者手动加入 Spring Security 的默认拦截栈, 并且配合 @WithMockUser 测试 API 的权限

  1. 自定义主键的 Generator

需要继承 SequenceStyleGenerator, 并且实现自己的 generated 方法. 并通过 @GenericGenerator 注册在 Entity 上, 之后就可以通过 @GeneratedValue 的 genetator 属性来使用自定义的主键生成策略.

  1. Lombok 与 JPA 整合使用时的坑.

分布式锁和事务

发表于 2019-08-04 更新于 2019-08-17

理解

分布式锁的场景

在微服务中同一个 service 有多个 application, 为了控制这多个 application 对相同的 resource 进行访问, 所采用的一种措施.

例如, 现在你的系统中有一个 order service, 同时这个 order service 启动了5个相同 order application. 它们都需要访问 order database. 这个时候如果不对这个5个 order application 进行加锁限制资源的访问, 那么就会出现诸如重复添加等问题.

分布式事务的场景

而分布式事务是指不同 service, 不同 resource 之间的所需要满足的数据 CAP 又或者 BASE. 所采用的一种措施.

例如, 你的 customer service 依赖于 order service. 而 customer service 拥有 customer database, order sevice 拥有 order database.

一般在微服务中, 每个 service 只能操作自己对应的 resource.

这个时候如果 customer service 在操作自己的 customer database 的同时, 还要要通过 order service 对 order database 进行操作, 就需要引入分布式事务来保证数据的最终一致性.

最佳实践

  • 分布式锁, 最好采用ZooKeeper
  • 分布式事务, 最好采用MQ
