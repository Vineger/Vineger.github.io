kubernetes

发表于 2019-08-10

Pod

kubernets 的最小部署单位. 抽象级别的操作系统. 定义了一组互相依赖的容器, 它们之间可以通过 localhost 或者是本地文件的方式进行数据交互.

这个抽象的出现是为了解决一组强依赖的部署对象. 例如, 日志应用, 普通应用之间就属于一种强依赖, 一般我们希望将这两个应用部署在同一台物理机器上. 而 Pod 的引入就是为了解决这个问题.

Service

访问集群中的 Pod 的一层网络代理, 具有负载均衡的功能.

Pod 在集群中的 IP 随着 Pod 的更新等操作, 不会一直绑定一个值. 这个时候就需要对外提供一层不变的 IP 供其它的 Pod 或者用户访问对应的 Pod. 可以暂时简单理解为对 Pod 做了一层反向代理.

Volumes

与 Docker 中的 Volumes 类似. 只是在 Kubernetes 中 Volumes 是与 Pod 绑定的. 且支持的类型更加丰富

Master & Node

  • Master 指的是 kubernetes 中的调度节点.
  • Node 指的是 kubernetes 中的物理节点.

Type of Vlomes

configMap

类似于配置中心. 通过文件将保存在 kubernetes 中的 key-value 注入 container 中.

secret

私有安全信息. 将保存在 kebernetes 中对应的安全信息通过文件挂载的方式注入 container. (注意, 保存在 kubernetes 中的 secre 信息必须是经过加密的.)

others

  • minikube, kubeadm 不可用于生产环境中.

Questions

junit5

发表于 2019-08-05 更新于 2019-08-10
  • 引入org.junit.jupiter:junit-jupiter:${version}就可以使用junit5. 注意要使用对应junit5包下的@Test等注解才有效.
  • @ParameterizedTest 与 @ValueSource 组合使用, 实现参数化测试.
    • @NullAndEmptySource 提供Null和Empty的参数输入
    • @EnumSource 将某个枚举类作为输入
    • @MethodSource 将某个静态方法的Stream类型的返回值作为参数输入
    • @CsvFileSource 允许从文件中读取参数化的输入
  • @Disable("{The Message about why you disable this Test}") 可以禁止某一测试的运行
  • @EnabledIfSystemProperty 通过系统变量判断, 测试是否要执行
  • @Order 安排测试运行的次序, 在Repository层的测试中或许很有用.
  • @RepeatedTest 标注测试重复的次数, RepetitionInfo获取每次重复的信息.

分布式锁和事务

发表于 2019-08-04

理解

分布式锁的场景

在微服务中同一个service有多个application, 为了控制这多个 application 对相同的 resource 进行访问, 所采用的一种措施.

例如, 现在你的系统中有一个order service, 同时这个order service 启动了5个相同 order application. 它们都需要访问order database. 这个时候如果不对这个5个 order application 进行加锁限制资源的访问, 那么就会出现诸如重复添加等问题.

分布式事务的场景

而分布式事务是指不同service, 不同resource之间的所需要满足的数据CAP又或者BASE. 所采用的一种措施.

例如, 你的customer service依赖于order service. 而customer service拥有customer database, order sevice 拥有 order database.

一般在微服务中, 每个service只能操作自己对应的resource.

这个时候如果customer service在操作自己的customer database的同时, 还要要通过order service对order database进行操作, 就需要引入分布式事务来保证数据的最终一致性.

最佳实践

  • 分布式锁, 最好采用ZooKeeper
  • 分布式事务, 最好采用MQ
